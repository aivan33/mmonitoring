# FaradaIC

**Use case:** `report` — monthly reporting pack for the management
meeting.
**Cadence:** monthly.
**Currency:** EUR. Germany + Serbia entities are consolidated and
FX-converted upstream in the master accounting workbook — the pipeline
sees a single consolidated view.

## What goes in

Each month, the following land in `clients/farada/raw/`:

| File | What it is | Source |
|---|---|---|
| `mr_<YYYY-MM>.xlsx` | Master accounting workbook ("MR"), DATEV-derived. Contains 10 sheets — consolidated P&L / CF / BS plus the upstream BWA, Trial Balance, Serbia entity, etc. | Bookkeeper's monthly delivery |
| `taxonomi_act_<YYYY-MM>.xlsx` (prior month) | The structured month-by-month canonical view, carried forward from the previous run | Generated by the previous month's pipeline run |
| `taxonomi_bp_2026.xlsx` | 2026 budget — all three scenarios (Pessimistic / Realistic / Optimistic) | Annual budgeting cycle |
| `taxonomi_bp_26nocap.xlsx` | Budget alternative without capitalization-policy adjustments | Same — used for sensitivity |
| `bookkeeping_<YYYY-MM>.xls` | Raw bookkeeping export | Bookkeeper — currently parked, not consumed by the pipeline |
| `model_no_cap_<YYYY-MM>.xlsx` | Internal projection model | Same — parked |

The previous month's deliverable PDF lives in `clients/farada/reference/`.
Open it next to `commentary.md` to confirm the structure mirrors.

The full MR sheet layout — which sheets are consumed, which are parked
for a deeper-mapping next iteration — is in
[`MR_LAYOUT.md`](MR_LAYOUT.md). The MR-row → canonical-key mapping is
in [`mapping.yaml`](mapping.yaml) (with sign-off from accounting).

## What happens

1. **Extract** the new month's column from the MR's P&L / CF / BS
   sheets and append it to the prior `taxonomi_act_*.xlsx`. The output
   is byte-stable apart from the new column — formatting and prior
   values are preserved (we copy-and-modify, never write-from-scratch).
2. **Reconcile** prior months between the MR and the canonical
   taxonomi to catch data-prep drift. Known intentional deltas
   (whitelisted in `mapping.yaml`'s `known_discrepancies`) don't
   re-flag; everything else surfaces under "New" for review before
   the rest of the pipeline runs.
3. **Load** the freshly-populated taxonomi into SQLite alongside the
   budget.
4. **Compute variance** — Actual vs Realistic budget, month-on-month,
   YTD — for IS / CF / BS. Materially-large variances (over thresholds
   in `config.yaml`) are flagged for management discussion.
5. **Draft commentary** mirroring the structure of the prior month's
   deliverable. Key figures are interpolated; flagged variances get a
   stub bullet with the numbers and a draft question. The user / a
   follow-up LLM session writes the prose.

> Phases 4 and 5 (variance, commentary) are stubbed today —
> `build_report.py --variance-only` and `--commentary-only` raise a
> clear `NotImplementedError` pointing at the future task.

## What comes out

`clients/farada/raw/taxonomi_act_<YYYY-MM>.xlsx` — the new structured
canonical file (carries forward as the next month's prior).

`clients/farada/reports/<YYYY-MM>/`:

- `reconcile.md` — prior-month deltas between MR and canonical
  (flags "New" deviations; "Known" deltas are documented in mapping.yaml)
- `variance.md` + `variance.csv` — tables of Actual vs Budget / MoM /
  YTD with flags on materially-large variances *(future)*
- `commentary.md` — Markdown outline mirroring the prior period's
  PDF structure, with key figures and flagged discussion items
  pre-populated *(future)*
- `checklist.md` — discussion items + carry-over topics *(future)*

## What to sanity-check each month

- **Spot-check 5+ rows of the new taxonomi against the MR.** Numbers
  in the new month's column must match the corresponding cell in the
  MR's P&L / CF / BS sheets exactly.
- **`reconcile.md` "New" section is empty.** Anything under "New"
  means either a real data-prep error (fix the source) or an
  intentional accounting change that needs adding to
  `mapping.yaml`'s `known_discrepancies` for next month.
- **MR-row layout hasn't shifted.** If the loader logs a
  `label-row mismatch`, the bookkeeper reordered rows in the MR.
  Update `mapping.yaml`'s `mr_row` indices to match, sign off again.
- **Carry-over topics in `config.yaml`** are still the right ones —
  prune them as items get resolved, add when the management meeting
  surfaces something new.

## Running the pipeline

See [`onboarding.md`](onboarding.md) for the FaradaIC-specific
engineer workflow (setup checklist + monthly runbook), or
[`docs/onboarding-report.md`](../../docs/onboarding-report.md) for
the general report-onboarding flow. TL;DR:

```bash
# Full pipeline
uv run python scripts/build_report.py farada <YYYY-MM> --all

# Or one phase at a time, with a review checkpoint between
uv run python scripts/build_report.py farada <YYYY-MM> --extract-only
uv run python scripts/build_report.py farada <YYYY-MM> --variance-only      # future
uv run python scripts/build_report.py farada <YYYY-MM> --commentary-only    # future
```
